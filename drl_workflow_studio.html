How to read this diagram (training vs inference, 42-d state, Data tables)

1. Two pipelines (the two colored bands)

Top — M1 → artifacts: Learn from data → replay buffer → train SAC → save agent.pt (+ JSON metadata). This is training.

Bottom — M2: Load agent.pt → safe agent → hybrid optimizer. This is inference (running the policy), not training. The edge from the checkpoint node down to load_sac_for_inference means: the file you trained is the one you load for serving.

2. What “42 dimensions” means

It is not “42 columns” in the small Input/Output tables. It means there is one field, state_vector, whose type is float32[42]: one array of 42 numbers in a fixed order (42 slots).

The large 42-row index list in the Data tab (when you scroll) names each slot (0 = ctr, …, 41 = constraint norms), matching CampaignState in drl/state_action.py.

3. What the Input / Output tables are

They are not a dump of every BigQuery row. They are a schema + sample: field names, types, example values, source, and where data flows next. “2 rows” in Input means two input fields (e.g. campaign_id, run_id), not two columns for the whole system.

4. “Where the 42-d output is used” (on MockCampaignEnv)

The state_vector (42 numbers) and reward are packaged into transitions (s, action, reward, next state, done) and sent to the Replay buffer node — the next box in the graph.

5. One-line summary

Training writes a checkpoint; inference loads it. 42-d = one state as 42 ordered features. The small tables describe names and types; the long index table explains what each position means.
